# 目的と最大の課題

以前のアプローチ（再帰によるDFS＋メモ化）では、状態空間の爆発によりメモリ上限（Out of Memory）に達してしまいます。
これを解決するため、再帰を一切使用せず、層化キャッシュを用いた「Bottom-Up型 動的計画法（DP）」を実装してください。

# 前提条件

- 52枚のカードを「スート(4) × ランク(A,K,Q,J,x)」の20バケットに分割して処理する。
- 判明しているカード（Known cards）は事前に除外しておく。

# コアエンジンのアルゴリズム設計（厳密な実装要件）

以下の4つのメモリ削減・高速化テクニックを必ず実装に組み込んでください。

1. Bottom-Up DP（層化キャッシュ）の採用:
    - 再帰関数を使わないでください。
    - `dp_prev = { initial_state: 1 }` からスタートし、バケットごとに `dp_curr = {}` を作成。
    - バケットの処理が終わるごとに `dp_prev = dp_curr` と更新し、古いメモリを解放してください。これにより同時に保持する状態数は「1バケットの横幅」のみになります。

2. 状態（State）の極限までの次元削減:
    - 状態を表すタプルには、「ユーザーが制約（HCPやシェイプ）を指定した要素」だけを動的に含めてください。
    - 例: EとWにHCPの指定がない場合、EとWのHCPは状態（キー）から除外する。
    - Wの残り枚数は `(全体の残り - N - S - E)` で求まるため、絶対に状態に含めない。

3. バケットの評価順序の最適化（Early Pruning）:
    - バケット（0〜19）を処理する順番を静的ではなく、条件に合わせてソートしてください。
    - 「アナー（A, K, Q, J）」のバケットを先に処理し、「x（スモール）」を後にする。これによりHCPの超過による枝刈り（Pruning）が序盤で大量に発生します。
    - 特定のスートに厳しい枚数制限がある場合、そのスートのバケットも優先して処理するロジックを入れてください。

4. 状態の遷移と組み合わせ計算:
    - 現在の状態（`dp_prev` のキー）から、現在のバケットのカード `C` 枚を N, S, E, W に分配する。
    - 分配時の組み合わせ数は多項係数 `math.comb` を用いて算出し、遷移前の組み合わせ数に乗算して `dp_curr` に加算する。
    - 遷移先の状態が、制約の「上限」を超えた場合、または残りのバケットをどう配っても「下限」に届かないことが確定した場合は、その状態を破棄（枝刈り）する。
